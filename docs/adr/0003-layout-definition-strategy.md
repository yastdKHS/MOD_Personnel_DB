# 0003. PDFレイアウトの外部データ定義化

## ステータス
Accepted

## コンテキスト

防衛省の人事発令PDFの様式（レイアウト・列構成・見出し表現）は、10年という時間軸では確実に変化する（過去の官報・公表資料でも様式変更は繰り返し発生している）。パースロジックにレイアウト依存の情報（座標、見出し文字列、列順）をハードコードすると、様式変更のたびにコード本体を変更する必要が生じ、以下の問題を招く。

- 変更のたびにコードレビューの負荷が高く、バグ混入リスクが上がる
- 過去の様式に対する再現性（同じPDFから同じ結果が得られること）が壊れやすい
- 「どの様式にどう対応しているか」が実装の中に散らばり、見通しが悪くなる

## 決定

- PDFレイアウトごとの構造情報（列位置、見出しパターン、様式の適用期間等）を `layouts/` 配下に**データとして**定義する（コードとして実装しない）。
- パース処理（`src/`）は、レイアウト定義を解釈する汎用エンジンとして実装し、様式固有のロジックを持たない。
- 新しい様式への対応は、原則として `layouts/` への追加のみで完結させることを目標とする。
- 各レイアウト定義は、対応する `sample_pdfs/` と `sample_outputs/` を伴い、ゴールデンファイルテストで検証する（[ADR-0007](0007-golden-file-testing.md)）。

## 検討した代替案

- **様式ごとにパーサークラスを実装（コードで表現）**: 柔軟性は高いが、レイアウトの違いという「データ的な差異」をコードの分岐として表現することになり、長期的な見通しの悪化・テストの複雑化を招く。今回は見送るが、レイアウト定義だけでは表現しきれない極端な例外様式が出た場合の逃げ道として、限定的な併用を将来検討する余地は残す。
- **PDFレイアウトを都度LLMに解釈させる**: 再現性・監査可能性に欠けるため、決定的（deterministic）なコアパイプラインの方針とは相容れない。決定的な処理で対応できない場合の補助的手段としてのみ将来検討し得る。

## 結果（トレードオフ）

- レイアウト定義のフォーマット自体の設計（`layouts/` のスキーマ）は慎重に行う必要があり、拡張性を欠くと結局コード側での例外処理が増える。フォーマットの詳細設計は別途実装時に検討する。
- レイアウト定義は非エンジニアでも理解しやすい形式（例: YAML/JSON）を目指し、ドメイン知識を持つがコードは書かない担当者でもレビューできるようにする。
