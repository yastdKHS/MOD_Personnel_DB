# 0007. ゴールデンファイルテスト戦略

## ステータス
Accepted

## コンテキスト

PDFパースのような「入力の微妙な違いが出力に大きく影響しうる」処理は、単体テストだけでは回帰（リグレッション）を検知しづらい。特に、あるPDF様式向けの修正が、別の様式の抽出結果を壊すケースを早期に検知する必要がある。

## 決定

- `sample_pdfs/` に、様式（era）ごとの代表的なサンプルPDFを格納する。
- `sample_outputs/` に、各サンプルPDFに対する期待される抽出・正規化結果（ゴールデンファイル）を格納する。
- `tests/golden` で、`sample_pdfs/` の各ファイルをパイプラインに通した結果が `sample_outputs/` と一致することを検証する。
- ゴールデンファイルの更新は「テストを通すための作業」ではなく「正解を変更する行為」として扱い、更新時はPRで変更理由を明記する（`CLAUDE.md` / `CONTRIBUTING.md` 参照）。

## 検討した代替案

- **ゴールデンファイルを持たず、フィールド単位のユニットテストのみで担保**: 個々のロジックの正しさは検証できるが、レイアウト定義全体としての整合性（実際のPDFを最後まで正しく処理できるか）を保証できない。両者を併用する方針とし、ゴールデンファイルはユニットテストの代替ではなく補完と位置づける。

## 結果（トレードオフ）

- サンプルPDFと期待出力のメンテナンスコストが発生するが、新しい様式に対応するたびに既存様式の処理が壊れていないことを機械的に保証できる利点の方が大きい。
- サンプルPDFの選定基準（何を代表例とするか）は [`sample_pdfs/README.md`](../../sample_pdfs/README.md) で定める。
