# 0029. 公開成果物の完全性保証と監査ログ方針

## ステータス
Accepted

## コンテキスト

[ADR-0026](0026-security-policy.md)はセキュリティ全般の方針（秘匿情報管理、依存ライブラリ脆弱性対応、外部由来PDFの解析リスク）を定めたが、以下の3点は範囲外として未決定のまま残されていた（Security Architecture設計、`docs/security.md`の作成過程で判明）。

1. **公開成果物（JSON/CSV/Parquet）の改ざん検知**: `exports.checksum`（[`docs/database/schema.md`](../database/schema.md)）は既に存在し、内容ハッシュ（SHA-256）による破損検知はできる。しかしチェックサム自体がファイルと同じ経路（FTP等）で配布される場合、経路上で両方を同時に差し替えられると検知できない。「誰が作ったデータか」を第三者が検証できる仕組み（改ざん**検知**を超えた**真正性**の保証）が欠落していた。
2. **GitHub Actionsのハードニング**: `.github/workflows/ci.yml`は`actions/checkout@v4`のようにタグ参照でActionsを利用している。タグは書き換え可能であり、サプライチェーン上のリスク（[ADR-0026](0026-security-policy.md)が触れる依存ライブラリの脆弱性対応と同種だが、Actions自体は対象外だった）になりうる。また、各ワークフローに与える`GITHUB_TOKEN`権限の最小化方針も未決定だった。
3. **横断的な監査ログの要件**: `ReviewHistory`（[`docs/review/domain.md`](../review/domain.md#reviewhistory)）や`jobs`テーブル等、ドメインごとの来歴・監査証跡は既に個別に設計されているが、「セキュリティ上重要な操作（Secretの利用、Gold Databaseへの書き込み、公開・Export操作）を横断的に監査可能にする」という要件そのものは、どのドメインの責務としても明示的に定められていなかった。

## 決定

- **署名**: 公開成果物（JSON/CSV/Parquet、[ADR-0022](0022-export-policy.md)）の`exports.checksum`に対して、Ed25519等の実績ある非対称鍵暗号方式による**デタッチド署名**を追加する。署名鍵は`production`環境専用のGitHub Secrets（[`docs/configuration.md`](../configuration.md#secret管理)）として保管し、公開鍵はリポジトリ内に平文で公開する。これにより、公開成果物の配布経路（FTP等）が信頼できない場合でも、公開鍵を保持する第三者が真正性を検証できる。具体的な署名ツール・アルゴリズムの確定は、[ADR-0026](0026-security-policy.md)が依存ライブラリ選定を実装時に委ねているのと同様、実装着手時に選定する（本ADRは「署名する」という方針と鍵管理の原則を決定するに留める）。
- **GitHub Actionsのハードニング**: サードパーティ製Actions（`actions/`名前空間以外を含む）は、可変なタグではなく**コミットSHAへのピン留め**を既定とする。GitHub公式の`actions/*`についても、将来的に同様のピン留めへ移行することを推奨する（既存の`ci.yml`は実装フェーズで段階的に移行し、本ADRをもって直ちに全面書き換えを強制しない）。各ワークフローの`GITHUB_TOKEN`権限は、既定の広い権限ではなく、ワークフローが必要とする最小権限（例: `contents: read`のみ）を明示的に指定する。
- **監査ログ**: セキュリティ上重要な操作（Secretへのアクセスが発生する実行、Gold Databaseへの書き込みを伴うReview承認、公開・Export・FTP送信）を、既存のドメイン別監査証跡（`ReviewHistory`, `jobs`, GitHub Actionsの実行ログ）から横断的に参照可能な形で扱うことを、セキュリティ上の要件として明示する。新たな専用テーブルは設けず（[ADR-0014](0014-development-discipline.md)の過剰設計回避の精神に従う）、既存の来歴データの集合を「監査ログ」という一つの機能要件として捉え直す。詳細は[`docs/security.md`](../security.md#audit-log)を参照。

## 検討した代替案

- **署名を導入せず、チェックサムのみで運用を続ける**: チェックサムは破損検知には有効だが、配布経路自体が侵害された場合の真正性検証はできない。公的機関の人事情報を扱うプロジェクトとして、10年という運用期間中に配布経路が信頼できなくなるリスクをゼロと見なすことはできないと判断し、署名を追加する方針とした。
- **監査ログ専用の新規テーブルを設計する**: `jobs`・`review_changes`・`learning_dataset`等、既に監査証跡としての役割を持つテーブルが複数存在する状態で、さらに専用テーブルを追加すると、どちらが正の記録かが曖昧になる（[ADR-0006](0006-pipeline-provenance.md)の来歴管理の思想と衝突する）。既存データの集合として扱う方針とした。

## 結果（トレードオフ）

- 署名鍵の管理（生成・保管・ローテーション・紛失時の対応）という新たな運用負荷が生じる。鍵の紛失は「過去の署名を検証できなくなる」という不可逆な影響を持つため、鍵のバックアップ・ローテーション手順は実装時に`docs/operations/`のRunbookとして必ず具体化する。
- GitHub ActionsのコミットSHAピン留めは、Actionsの自動更新（Dependabot等によるバージョン追従）の恩恵を弱め、手動または半自動でのピン留め更新作業が必要になる。これは意図的なトレードオフであり、サプライチェーンの検証可能性を優先する。
- 監査ログを既存データの集合として扱う方針は、新規実装コストを避けられる一方、実装時に「横断的に検索・参照するための最小限のクエリ層」を`services/`等に用意する必要がある。この点は`docs/security.md`で補足する。
