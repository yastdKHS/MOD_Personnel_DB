# 0026. セキュリティポリシー

> 本ADRの決定を脅威モデルとして構造化し、各対策領域を具体化した設計は[`docs/security.md`](../security.md)（Security Architecture）を参照。署名・GitHub Actionsハードニング・監査ログの追加決定は[ADR-0029](0029-export-integrity-and-audit-log-policy.md)を参照（本ADRの内容は変更しない）。

## ステータス
Accepted

## コンテキスト

[ADR-0008](0008-data-ethics-policy.md)は個人情報・データ倫理（収集範囲の限定）を定めるが、これは「データの取り扱い」に関する方針であり、システムとしての**セキュリティ**（秘匿情報管理、依存ライブラリの脆弱性対応、外部由来のPDFファイルを解析することに伴うリスク）は別の関心事として未決定だった（[Gap Analysis](gap-analysis.md#security-policy)参照）。`.pre-commit-config.yaml`にgitleaksは既に組み込まれているが、これは個別の実装上の施策であり、包括的な方針を定めるADRは存在しなかった。

## 決定

- **秘匿情報管理**: APIキー・認証情報等はリポジトリに含めず、実行環境（GitHub Actions Secrets等、[ADR-0019](0019-workflow-orchestration.md)）の秘密情報管理機構を用いる。誤コミット検知として`.pre-commit-config.yaml`のgitleaksを維持する。
- **依存ライブラリの脆弱性対応**: 依存関係（`pyproject.toml`、[ADR-0001](0001-python-packaging.md)）の脆弱性を、CI（[ADR-0010](0010-ci-cd-and-publish-strategy.md)）に自動スキャン（例: `pip-audit`または同等ツール）として組み込み、既知の重大な脆弱性を持つ依存関係の追加・放置を防ぐ。
- **外部由来PDFの解析リスク**: PDFパースライブラリの選定時（実装着手時）に、既知の脆弱性（圧縮爆弾、リソース枯渇、パーサーの脆弱性に起因する任意コード実行等）に対する実績・メンテナンス状況を選定基準に含める。Document Analyzer段階（[ADR-0011](0011-fixed-core-pipeline.md)）にメモリ・実行時間の上限を設け、異常なPDFがパイプライン全体を停止させないようにする。
- 本プロジェクトの性質上、認証・認可の仕組みは、公開成果物が静的ファイル配信（[ADR-0025](0025-deployment-strategy.md)）である限り最小限とし、主眼は「意図しない情報の混入・拡散の防止」（[ADR-0008](0008-data-ethics-policy.md)）に置く。

## 検討した代替案

- **セキュリティ対応を都度その場で判断し、方針を明文化しない**: 10年運用・複数担当者・AIコーディングエージェントによる開発という本プロジェクトの性質上、判断基準がぶれるリスクが高いため、最小限でも明文化する方針とした。

## 結果（トレードオフ）

- 具体的なツール（`pip-audit`等）の選定・CI組み込みは実装時に行うが、「何を・なぜ守るか」という方針を先に固定することで、実装時の判断のブレを防ぐ。
- PDFパースライブラリへの制約（実績・メンテナンス状況を選定基準に含める）は、ライブラリ選定の自由度を狭めるが、外部由来ファイルを継続的に処理するシステムとして必要なトレードオフと判断した。
