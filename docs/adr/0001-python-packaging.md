# 0001. Pythonパッケージング・ビルドバックエンドの選定

## ステータス
Accepted

## コンテキスト

本プロジェクトは10年以上の保守を前提とする。Pythonのパッケージング/ビルドツールは数年単位で流行が移り変わる領域（setuptools → Poetry → PDM → uv 等）であり、特定ツールへの過度な依存はリスクになる。一方で、依存関係の再現性（同じ入力から同じ環境を再構築できること）も長期運用には不可欠。

## 決定

- プロジェクトメタデータ・依存関係の定義は `pyproject.toml`（PEP 621）に一本化する。
- ビルドバックエンドは **setuptools** を採用する。
- パッケージレイアウトは `src/` レイアウトを採用する（`src/mod_personnel_db/`）。
- インストール・実行は標準の `pip` で行える状態を維持する。特定の高速ツール（uv 等）は開発体験向上のための任意選択とし、`pyproject.toml` の標準性を崩さない。
- 依存の再現性が必要な場面（本番実行環境等）では `pip-tools`（`pip-compile`）によるロックファイル生成を検討する。

## 検討した代替案

- **Poetry**: 独自の依存解決・ロック機構が便利だが、`pyproject.toml` の一部が非標準（`[tool.poetry]`）であり、ツール依存が強い。10年後の存続不確実性を考慮し見送り。
- **uv**: 高速で近年急速に普及しているが、比較的新しいツール（単一ベンダー主導）であり、長期の安定運用実績がまだ確立していない。ただしPEP621準拠のため、将来的にフロントエンドとして併用することは妨げない。
- **hatchling**: 軽量でモダンだが、setuptoolsほどの長期実績・エコシステム定着はまだない。

## 結果（トレードオフ）

- setuptoolsは「枯れている」ことが最大の利点であり、先進的な機能（高速な依存解決等）はない。開発体験は他ツールに劣る場面があるが、`pyproject.toml` が標準準拠である限り、将来的にビルドバックエンドのみを差し替えることは比較的容易。
- 依存関係を追加する際は、`dev` 以外の実行時依存を最小限に保つ方針とする（依存が少ないほど、長期的な壊れにくさが高い）。
