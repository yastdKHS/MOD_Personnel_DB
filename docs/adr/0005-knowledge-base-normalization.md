# 0005. ドメイン知識ベースによる名寄せ・正規化戦略

## ステータス
Accepted

## コンテキスト

人事発令データには、階級呼称の変更、組織の改称・改編、氏名の表記ゆれ（異体字・旧字体等）といった「コードのロジックではなく、時間とともに増えていく知識」が大量に存在する。これをコード内の条件分岐（if文の羅列）で表現すると、以下の問題が生じる。

- ドメイン知識を追加するたびにコード変更・レビューが必要になり、変更コストが高い
- ロジックとデータが混在し、正しさの検証（このマッピングは正しいか）がコードレビューに埋没する
- ドメイン知識を持つがPythonは書けない担当者が貢献しにくい

## 決定

- 階級名・組織名・改称履歴・氏名の異体字対応等のドメイン知識は、`knowledge/` 配下に構造化データとして保持する。
- `src/` の正規化ロジックは、`knowledge/` のデータを参照する汎用エンジンとして実装し、個別のマッピングをコードにハードコードしない。
- `knowledge/` の変更は、通常のコード変更と同様にPRレビューを経る。ただし変更内容は「ロジックの正しさ」ではなく「データの正しさ（出典・根拠があるか）」の観点でレビューする。

## 検討した代替案

- **正規化ルールをコードの辞書リテラル・条件分岐で表現**: 実装は簡単だが、[ADR-0003](0003-layout-definition-strategy.md) と同様の理由で長期的な保守性を損なうため見送り。
- **外部の統制語彙・オントロジーサービスを利用**: 防衛省組織・自衛官階級に特化した外部語彙は一般に存在しないため、自前でのナレッジベース構築が現実的と判断した。

## 結果（トレードオフ）

- `knowledge/` のデータ品質がシステム全体の正確性を左右するため、変更管理（レビュー・出典明記）を軽視できない。運用コストは増えるが、正規化ロジックの複雑化を防ぐメリットの方が大きいと判断した。
- 正規化前の生データ（`raw_text` 等）を常に保持する設計（[データモデル](../data_model.md)参照）と組み合わせることで、`knowledge/` の誤りが発覚した場合も再正規化により修正可能にする。
