# 0025. デプロイメント戦略

## ステータス
Accepted

## コンテキスト

[ADR-0004](0004-sqlite-as-datastore.md)はデータストアの選定を、[ADR-0010](0010-ci-cd-and-publish-strategy.md)はCI/CDのガバナンスを定めるが、本番運用のシステムが**どこで・どのように稼働するか**（常時稼働サーバー、コンテナ、サーバーレス、バッチ実行等）という実行環境そのものの選定は未決定だった（[Gap Analysis](gap-analysis.md#deployment-strategy)参照）。SQLite（単一ファイルDB）という選定は、複数プロセスからの同時書き込みを前提とする構成とは相性が悪く、デプロイ形態の選択肢を実質的に絞り込むため、[ADR-0004](0004-sqlite-as-datastore.md)と整合させた形で明示的に決定する必要がある。

## 決定

- 本番運用は、常時稼働サーバーを持たない**バッチ実行モデル**を採用する。[ADR-0019](0019-workflow-orchestration.md)で決定したGitHub Actionsのスケジュール実行上で、都度実行環境（GitHub Actions runnerの標準コンテナ、将来的には自前のコンテナイメージ）を起動し、処理完了後に終了する。
- SQLiteデータベースファイルは実行間で永続化する必要があるため、実行のたびに永続ストレージ（[ADR-0018](0018-pdf-registry-and-retention.md)と同様の外部ストレージ）から読み込み、処理後に書き戻す。複数の書き込みプロセスが同時に走らないよう、ワークフロー側の排他制御（同時実行の禁止）を設ける。
- 公開用の成果物（[ADR-0022](0022-export-policy.md)のJSON/CSV/Parquetエクスポート）は静的ファイルとして配信可能な形態を優先し、常時稼働のAPIサーバーは本方針の対象外とする。

## 検討した代替案

- **常時稼働サーバー（VM/コンテナサービス）を用意する**: 運用・監視・セキュリティパッチ適用等の継続的な保守コストが、バッチ実行方式に比べて高く、10年運用における保守負荷最小化の方針（[ADR-0001](0001-python-packaging.md)）に反するため見送った。
- **サーバーレス関数（AWS Lambda等）**: 実行時間制限やローカルファイル（SQLite）の扱いに制約があり、バッチ実行の延長として同等の効果をGitHub Actionsで得られるため、追加のクラウドベンダー依存を避ける方針とした。

## 結果（トレードオフ）

- 常時稼働サーバーがないため、リアルタイム性の高い用途（即時APIアクセス等）には対応できない。将来そのような要件が生じた場合は、本ADRをSupersedeする形で再検討する。
- SQLiteファイルの読み書きと排他制御をワークフロー側で管理する必要があり、実装時にはこの排他制御の確実性（多重実行防止）を重点的に検証する必要がある。
