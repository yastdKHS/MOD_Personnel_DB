# 0011. 中核処理パイプラインの固定化

## ステータス
Accepted（このADRの決定は設計上の不変条件であり、通常のADR以上に変更のハードルを高く置く。変更する場合は単なる新規ADRの追加ではなく、プロジェクトオーナーの明示的な承認を要する）

## コンテキスト

[ADR-0006](0006-pipeline-provenance.md) はパイプラインをステージに分割する方針を定めたが、各ステージの粒度・名称は「取得→抽出→パース→正規化→検証→格納→公開」という比較的粗い記述にとどまっていた。粒度が粗いままだと、実装者ごとに「抽出とパースの境界はどこか」の解釈がぶれ、結果として一つの巨大な関数・モジュールにロジックが集約されたり、ステージの責務が曖昧になったりするリスクがある。

10年運用において、パイプラインの段階構成そのものが実装のたびに揺れ動くことは、[ADR-0003](0003-layout-definition-strategy.md)・[ADR-0005](0005-knowledge-base-normalization.md) が目指す「変化の吸収先をlayouts/・knowledge/に閉じ込める」という設計の前提を崩す。段階構成自体は固定し、変化はデータ（layouts/・knowledge/）側で吸収する、という役割分担を明確にする必要がある。

## 決定

PDF解析の中核パイプラインを、以下の6段階に**固定**する。

```
Document Analyzer
  ↓
Layout Detector
  ↓
Section Parser
  ↓
Field Extractor
  ↓
Normalizer
  ↓
Validator
```

- この段階の**数・順序・名称を変更しない**。ステージの統合（例: Section ParserとField Extractorを1モジュールにまとめる）、分割、順序の入れ替え、新規ステージの挿入は行わない。
- 各段階は単一責務を持つ。ある段階が別の段階の責務を肩代わりする実装（例: Field ExtractorがLayout Detectorの役割を兼ねる）は設計違反として扱う。
- 新しいPDF様式・新しい例外への対応は、この6段階の構成を変えるのではなく、各段階が参照する `layouts/` / `knowledge/` のデータを増やすことで行う（[ADR-0012](0012-error-handling-priority-order.md)）。
- この中核パイプラインの前後には、取得（Fetch）・格納（Store）・公開（Publish）という外部ステージが存在するが、これらは中核パイプラインの一部ではなく、本ADRの固定対象に含まれない。

各段階の詳細な責務は [`docs/architecture.md`](../architecture.md) を参照。

## 検討した代替案

- **段階構成を「指針」として緩く保ち、実装の都合で調整可能にする**: 柔軟性は高いが、実装者・AIエージェントのセッションが変わるたびに構成が微妙に変わり、10年後には「今の構成が正か」を誰も説明できなくなるリスクが高い。今回は「固定して変化をデータ側に押し出す」方針を優先し、緩い方針は見送った。
- **ステージをより細かく分割する（例: Field Extractorをフィールド種別ごとに分割）**: 過度な分割は段階間の連携コストを増やし、「大きな関数の禁止」（[ADR-0014](0014-development-discipline.md)）はモジュール内の関数粒度で担保すれば十分と判断し、パイプライン段階そのものの細分化は見送った。

## 結果（トレードオフ）

- 将来、この6段階では表現しきれない根本的に異なる処理が必要になった場合（例えば人事発令PDF以外の全く異なる文書種別を扱う等）、本ADRの変更が必要になる。その際は通常のADR追加より高いハードル（プロジェクトオーナーの明示的承認）を課すことで、安易な変更を防ぐ。
- 段階構成が固定されることで、新しい担当者・AIエージェントは「今どの段階の作業をしているか」を常に明確に認識でき、責務混在によるバグを防ぎやすくなる。
