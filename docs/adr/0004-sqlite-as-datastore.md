# 0004. データストアとしてのSQLite採用

## ステータス
Accepted

## コンテキスト

データ規模は「防衛省の人事発令」という性質上、大規模ビッグデータではなく、長期間かけて蓄積される中規模の構造化データになる見込みである。10年運用において、データストアの運用負荷（サーバー管理、バージョンアップ対応、障害対応）を最小化したい。また、公開データとしての配布のしやすさも重要な要件になる。

## 決定

- 正データ（source of truth）は **SQLite** の単一ファイルDBとして管理する。
- 公開用途に応じて、SQLiteから CSV / Parquet 等の派生フォーマットをエクスポートする（公開ステージの責務）。
- スキーマ変更はマイグレーションスクリプトとして管理し、変更履歴を追跡可能にする（具体的なマイグレーションツールは実装時に選定）。

## 検討した代替案

- **PostgreSQL 等のサーバー型RDBMS**: 同時書き込み・大規模データには強いが、本プロジェクトの更新頻度（発令のたびのバッチ処理）に対しては過剰であり、10年間サーバーを維持運用するコスト・リスクの方が大きいと判断した。将来的にデータ規模やアクセスパターンが変わった場合は、SQLiteからの移行を別途ADRとして検討する。
- **CSV/JSONのみをデータストアとする（RDBMSを持たない）**: シンプルだが、整合性制約・関連の表現・クエリ能力に欠け、検証ステージの実装が困難になる。
- **DuckDB**: 分析用途に強いが、本プロジェクトの主用途はトランザクショナルな追記・更新であり、SQLiteの方が実績・エコシステムの成熟度で優位と判断した。

## 結果（トレードオフ）

- SQLiteは単一ファイルであるため、バックアップ・配布・監査（ファイルごとバージョン管理）が容易になる。
- 大規模な同時書き込みには向かないが、本プロジェクトのワークロード（バッチ処理中心）とは合致する。
- 将来データ量・利用形態が想定を超えて拡大した場合に備え、データアクセス層（`src/`）はSQLite固有の機能に過度に依存しない実装とする。
