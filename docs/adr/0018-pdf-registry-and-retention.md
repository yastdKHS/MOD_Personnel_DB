# 0018. PDF Registry・長期保管方針

## ステータス
Accepted

## コンテキスト

[ADR-0006](0006-pipeline-provenance.md)は「すべてのレコードは、どのPDFのどの記載から生成されたかを追跡できなければならない」という来歴管理の方針を定め、`docs/database/schema.md`の`pdfs`テーブルはそのメタデータ（`content_hash` / `source_url` / `published_date` / `file_path`）を保持する設計を持つ。しかし、以下は未決定のまま残っていた（[Gap Analysis](gap-analysis.md#pdf-registry)参照）。

- `file_path`が指す実PDFファイル本体を、実際にどこに・どのような構造で・どれだけの期間保管するか
- 取得に失敗した場合のリトライ方針
- ストレージ側の破損（bit rot）をどう検知するか

## 決定

- 実PDFファイル本体は、本リポジトリの外部にある専用ストレージに保管する（具体的な製品・サービスは実装時に選定するが、10年規模の可用性・バックアップ体制を満たすことを要件とする）。`sample_pdfs/`（リポジトリ内、[ADR-0007](0007-golden-file-testing.md)）とは別の実体である。
- 保管パスは、内容アドレス方式（content-addressable）のシャーディング構造とする: `<content_hash先頭2文字>/<content_hash次2文字>/<content_hash>.pdf`。ディレクトリ肥大化を避けるための慣例的な分割である。
- `pdfs.content_hash`のUNIQUE制約（`docs/database/schema.md`）を正とし、同一内容のPDFを異なるURLから複数回取得しても、`pdfs`テーブルには重複行を作らない。最初に確認された`source_url`のみを記録し、以降の再取得の試行は`jobs`テーブルの実行記録として残す。
- 取得失敗時は指数バックオフによるリトライを行い、上限回数（実装時に確定）を超えて失敗した場合は`jobs.status='failed'`として記録し、人手確認の対象とする。
- 保管中のファイルは定期的（少なくとも年1回）に再ハッシュ検証を行い、`pdfs.content_hash`と一致することを確認する。不一致が検出された場合はストレージ破損として扱い、原本の再取得または復旧を行う。

## 検討した代替案

- **PDF本体をSQLiteのBLOB列として格納する**: SQLiteはBLOBを扱えるが、単一ファイルDBが肥大化し、[ADR-0004](0004-sqlite-as-datastore.md)が意図する「バックアップ・配布の容易さ」という利点を損なう。DBはメタデータのみを保持し、実体は外部ストレージに置く現行設計を維持する。
- **取得のたびに無条件で新規`pdfs`行を作成する**: 同一内容の重複取得のたびにレコードが増え、[ADR-0006](0006-pipeline-provenance.md)が意図する「来歴の一意性」を損なう。`content_hash`による重複排除を維持する。

## 結果（トレードオフ）

- 実体ストレージの技術選定（具体的な製品）は実装時に持ち越すが、契約（内容アドレス方式、シャーディング、整合性検証の頻度）は本ADRで先に固定する。
- 同一内容が複数のURLから取得され得るという事実（`source_url`が変わる可能性）を、`pdfs`テーブル上では最初の1件のみで表現するため、URL変遷の完全な履歴が必要な場合は`jobs`の実行ログを別途参照する必要がある。
