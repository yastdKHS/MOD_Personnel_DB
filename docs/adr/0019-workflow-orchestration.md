# 0019. 実行オーケストレーション（Workflow）戦略

## ステータス
Accepted

## コンテキスト

`docs/database/schema.md`の`jobs`テーブルは、パイプライン実行の結果（開始・終了・処理件数）を記録する設計を持つが、その実行を**いつ・どうやって起動するか**（定期実行のトリガー、実行環境、失敗時の扱い）は決定されていなかった（[Gap Analysis](gap-analysis.md#workflow-engine)参照）。[ADR-0011](0011-fixed-core-pipeline.md)が固定した中核パイプライン（Document Analyzer〜Validator）に加え、その前後の取得（Fetch）・格納（Store）・公開（Publish）を含む一連の処理を、どの基盤で連結・実行するかという運用上の意思決定が欠落していた。

## 決定

- 定期実行のトリガーには、既にCI基盤として採用している GitHub Actions（[ADR-0010](0010-ci-cd-and-publish-strategy.md)）のスケジュール実行（`schedule: cron`）を用いる。新たな専用ワークフロー基盤（Airflow等）を追加で導入・保守するコストを避けるため。
- 各ワークフロー実行は`jobs`テーブルに1行記録される。中核パイプラインの各段階（[ADR-0011](0011-fixed-core-pipeline.md)）の失敗が、他のPDFの処理や後続のワークフロー全体の停止に波及しないよう、PDF単位で独立して成否を扱う。
- 実行の失敗に対するリトライは、まずGitHub Actions自体のリトライ機構（再実行）に委ね、恒久的に失敗する場合は人手対応（Issue化）とする。
- 処理量・実行時間がGitHub Actionsの制約（1ジョブあたりの実行時間上限等）を超える規模に達した場合は、専用ワークフローエンジンへの移行を、その時点で新規ADRとして検討する。本ADRはあくまで初期規模における選定である。

## 検討した代替案

- **専用ワークフローエンジン（Airflow, Prefect等）を最初から導入する**: 学習コスト・運用コスト（別サービスの監視・可用性維持）が、現時点の処理量（防衛省の発令PDF、不定期・小規模）に対して過剰と判断し、規模拡大時の見直し対象として保留した。
- **サーバー常駐でのcron実行**: [ADR-0025](0025-deployment-strategy.md)（デプロイメント戦略）と方針が重複・矛盾するため、常時稼働サーバーを前提としないマネージドスケジューラ（GitHub Actions）優先の方針に統一した。

## 結果（トレードオフ）

- GitHub Actionsには無料枠・実行時間の制約があり、将来のデータ量増加時には制約に達する可能性がある。その場合の移行判断基準（実行時間・頻度の閾値）は、実装・運用開始後に具体化する。
- ワークフロー定義（`.github/workflows/`）が、実質的な「パイプライン起動の正」になるため、[`docs/architecture.md`](../architecture.md)のパイプライン説明とワークフロー定義の整合を維持する運用上の注意が必要になる。
