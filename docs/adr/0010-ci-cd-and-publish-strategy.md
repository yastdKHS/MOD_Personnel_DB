# 0010. CI/CDと公開戦略

## ステータス
Accepted

## コンテキスト

長期運用プロジェクトでは、CIが形骸化（常に赤い、あるいは意味のあるチェックをしていない）すると、品質担保の仕組み自体が信頼されなくなる。また、データベースの「公開」は通常のソフトウェアリリースとは性質が異なり、公開後の訂正・撤回のコストが高い。

## 決定

- CI（`.github/workflows/ci.yml`）は、lint（ruff）・型チェック（mypy）・テスト（pytest、ゴールデンファイル含む）をPRごとに実行する。
- 実装コードが存在しない段階（現状）では、対象ファイルの有無を確認しチェックをスキップすることで、CIを恒常的に失敗させない。実装追加後は自動的にチェック対象へ移行する。
- データベースの「公開」は通常のコードデプロイとは別の意思決定ステップとして扱う。検証（Validate）ステージを通過したデータのみが公開候補になり、公開自体は明示的なリリース操作（自動マージによる即時公開はしない）とする。
- 公開したデータに事後的な訂正が入った場合、変更履歴（何がいつ、なぜ変わったか）を追跡可能な形で公開する（[ADR-0006](0006-pipeline-provenance.md) の来歴管理と対応）。

## 検討した代替案

- **mainへのマージと同時にDBを自動公開**: 実装の単純さは魅力だが、人事データという性質上、公開前に人手のチェックを挟める余地を残す方が安全と判断し、公開は別ステップとする方針とした。

## 結果（トレードオフ）

- 公開フローに人手のステップを残すことで、完全自動化に比べて公開頻度は落ちるが、誤ったデータが即座に公開されるリスクを抑える。
- 具体的な公開先・公開形式（静的サイト、API、データファイル配布等)は実装時に別途検討し、必要であれば新規ADRを起票する。
