# 0002. Lint / Format / 型チェックツールの選定

## ステータス
Accepted

## コンテキスト

複数人（および複数のAIエージェントセッション）が長期間にわたりコードを書くため、スタイルの一貫性とバグの早期検出を自動化する必要がある。ツールの学習コストと将来の保守性のバランスを取る必要がある。

## 決定

- Lint・フォーマット: **ruff** を採用する（flake8 + isort + black相当を単一ツールで代替）。
- 型チェック: **mypy** を `strict` 設定で採用する。
- 設定は `pyproject.toml` に一元化し、個別設定ファイル（`.flake8`, `setup.cfg` 等）を増やさない。
- pre-commit（`.pre-commit-config.yaml`）とCI（`.github/workflows/ci.yml`）で同一のチェックを走らせる。

## 検討した代替案

- **flake8 + black + isort の組み合わせ**: 実績は長いが、3ツールの設定・バージョン管理コストが高い。ruffが機能的に上位互換であるため見送り。
- **型チェックにpyright**: エディタ統合に強いが、CLIでの厳格モード運用実績・エコシステムはmypyの方が長い。将来的な併用は妨げない。

## 結果（トレードオフ）

- ruffは単一ベンダー（Astral）依存というリスクはあるが、出力（フォーマット済みコード）自体には特別なロックインがなく、将来別ツールへ切り替えても実害は小さい。
- `mypy strict` は初期の実装コストを上げるが、10年運用における「型のドキュメント化」効果を優先する。
