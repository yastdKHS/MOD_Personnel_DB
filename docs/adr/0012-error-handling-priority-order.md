# 0012. 未知パターンへの対応優先順位（Knowledge Base > Layout > 例外処理）

## ステータス
Accepted

## コンテキスト

PDFのパース・正規化を実装していると、「既存のロジックでは扱えない未知のパターン」に日常的に遭遇する（新しい階級表記、未知の組織名、想定外のPDF構造等）。このとき、最も安易な対応は `src/` のコードに `if` 分岐や正規表現、`try/except` による特殊対応を追加することである。しかし、これは [ADR-0003](0003-layout-definition-strategy.md)・[ADR-0005](0005-knowledge-base-normalization.md) が避けようとした「ドメイン知識・様式差異のコードへのハードコード」そのものであり、放置すると `src/` が特殊対応の集積地になり、長期的な保守性を損なう。

## 決定

未知パターンに遭遇した場合、以下の優先順位で対応する。番号が小さいほど優先する。

1. **Knowledge Base（`knowledge/`）への追加** — 値の表記ゆれ・別名・改称履歴等、「データとして既知にすれば解決する」問題は、まず `knowledge/` にエントリを追加することで対応する。正規表現による特殊対応より、常にこちらを優先する。
2. **Layout（`layouts/`）への追加** — PDLの構造・様式そのものに起因する問題（列位置が違う、見出しが違う等）は、新しい `layouts/<era_id>/` の追加、または既存レイアウト定義のデータとしての拡張で対応する。`src/` 側での例外処理（try/except による特殊分岐）より、常にこちらを優先する。
3. **`src/` 内の例外処理・正規表現による特殊対応** — 上記2つで表現しきれない場合の**最後の手段**とする。追加する場合は、なぜKnowledge BaseまたはLayoutで表現できなかったのかをコードコメントおよびPR説明に明記し、レビューで妥当性を確認する。

この優先順位は、[ADR-0011](0011-fixed-core-pipeline.md) で固定した中核パイプラインの段階構成を変えずに変化を吸収するための実践的な運用ルールである。

## 検討した代替案

- **都度、実装者の判断に委ねる**: 実装速度は上がるが、「とりあえずコードで対応する」が既定の選択になりがちで、`src/` が肥大化する。長期運用の観点から明示的な優先順位を定める方針とした。
- **例外処理を一切禁止する**: 理想的だが非現実的。PDFという外部由来の非構造データを扱う以上、Knowledge BaseやLayoutでも表現しきれない真の例外は必ず発生する。禁止ではなく「最後の手段」と位置づけることで、現実的な運用を可能にした。

## 結果（トレードオフ）

- Knowledge Base・Layoutへの追加は、コードでの分岐追加よりレビュー・データ整備の手間がかかる場合がある。それでも、データはコードよりレビューが容易で、非エンジニアの貢献者も参加できるという長期的な利点を優先する。
- `src/` 内の例外処理が増え始めた場合、それは「Knowledge BaseまたはLayoutの設計が実態に追いついていない」というシグナルとして扱い、コードレビューで指摘する。
