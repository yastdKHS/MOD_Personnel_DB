# 0023. Parserバージョニング方針

## ステータス
Accepted

## コンテキスト

`docs/database/schema.md`の`parser_versions`テーブルと、公開JSON形式の3層バージョン管理（[`docs/database/json_schema.md`](../database/json_schema.md#バージョン管理)：DBスキーマバージョン／データ生成バージョン／公開JSON形式バージョン）は既に設計済みである。しかし、データ生成バージョンそのもの（`parser_versions.code_version`）が**どのように採番されるか**（コミットハッシュか、リリースタグか）、**いつ新しい行が作られるか**（マージのたびか、リリースのたびか）は未決定だった（[Gap Analysis](gap-analysis.md#parser-versioning)参照）。この決定が曖昧なままだと、[ADR-0006](0006-pipeline-provenance.md)が要求する来歴追跡の粒度が実装者ごとにブレる。

## 決定

- `parser_versions.code_version`には、Gitのリリースタグ名（SemVer形式、例: `v1.3.0`）を用いる。個々のコミットハッシュではなく、意図的にタグ付けされたリリースのみを記録する。
- 新しい`parser_versions`行は、`main`ブランチへのリリースタグ付与をトリガーに、CI（[ADR-0010](0010-ci-cd-and-publish-strategy.md)）が自動的に作成する。手動INSERTは行わない（漏れのリスクを避けるため）。
- SemVerの増分規則を以下のように定める。
  - **MAJOR**: 既存レコードの再現性を壊す変更（過去の入力に対する抽出結果が変わりうる変更）
  - **MINOR**: 中核パイプライン（[ADR-0011](0011-fixed-core-pipeline.md)）の出力形式に追加が生じる、後方互換な変更（新フィールドの追加等）
  - **PATCH**: 出力に影響しないバグ修正・内部リファクタリング
- リリースタグは、来歴の恒久的な参照点であるため、削除・強制上書き（`git tag -f`での付け替え等）を禁止する（`AGENTS.md`の破壊的操作禁止規定と整合）。

## 検討した代替案

- **コミットハッシュをそのまま`code_version`とする**: 意図しない中間コミット単位の粒度になり、「これはリリース版か作業中の版か」の区別がつかない。タグベースの方が運用上明確であり、リリースという人為的な区切りと来歴の粒度を一致させられると判断した。
- **リリースのたびに手動で`parser_versions`へINSERTする**: 手動操作は漏れのリスクを伴うため、CIによる自動化を方針とした。

## 結果（トレードオフ）

- リリースの粒度（いつタグを打つか）の判断が、そのまま来歴の粒度になる。頻繁すぎるタグ付けは`parser_versions`の肥大化を招くため、リリース基準は`CONTRIBUTING.md`のPRプロセスと連動する形で運用時に定める。
