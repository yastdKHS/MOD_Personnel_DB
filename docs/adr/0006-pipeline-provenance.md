# 0006. パイプライン段階分割と来歴（Provenance）管理

## ステータス
Accepted

## コンテキスト

10年以上運用する中で、抽出・正規化のミスは必ず発生する。重要なのは「ミスをゼロにすること」ではなく「ミスが起きたときに、影響範囲を特定し、正しく修正できること」である。そのためには、各データがどのPDF・どの処理過程を経て生成されたかを常に追跡できる必要がある。

## 決定

- パイプラインを [取得 → 抽出 → パース → 正規化 → 検証 → 格納 → 公開] の明確なステージに分割する（詳細は [`architecture.md`](../architecture.md)）。
- 各ステージの出力は、入力元（前ステージの出力、または元PDFのハッシュ）への参照を保持する。
- 最終的なDBレコードは、根拠となる `source_documents`（PDFの取得日時・ハッシュ・URL）に必ず紐づく。根拠を持たないレコードは存在させない。
- 検証（Validate）ステージで異常と判定されたデータは、サイレントに破棄せず、要確認キューとして保持し、人手のレビュー対象とする。

## 検討した代替案

- **取得からDB格納までを単一プロセス・単一トランザクションとして実装**: 実装は単純化するが、途中経過の検証・デバッグが困難になり、障害時の原因特定コストが跳ね上がる。長期運用における可観測性を優先し見送り。

## 結果（トレードオフ）

- ステージ分割・来歴管理の分だけ初期実装コストは増える。
- 一方で、後年「このデータはなぜこの値になっているのか」を追跡できることは、公的データを扱うプロジェクトの信頼性に直結するため、必須要件として扱う。
