# 0014. 開発規律（設計品質優先・1PR1責務・関数サイズ制限）

## ステータス
Accepted

## コンテキスト

10年以上の運用を前提とするプロジェクトにおいて、実装速度を優先する開発規律は、短期的な進捗と引き換えに長期的な保守性を損なう。特に、複数の変更を含む大きなPull Request、責務が肥大化した関数は、後から「なぜこうなっているか」を読み解くコストを指数関数的に増やす。本プロジェクトでは、実装速度より設計品質を優先することを明文化し、これを可能な限り自動チェックで担保する。

## 決定

以下を本プロジェクトの開発規律として定める。

1. **実装速度より設計品質を優先する**。レビューで「動くが設計原則に反する」変更は、動作の正しさに関わらずマージしない。
2. **設計を壊すコードを禁止する**。[ADR-0003](0003-layout-definition-strategy.md)・[ADR-0005](0005-knowledge-base-normalization.md)・[ADR-0011](0011-fixed-core-pipeline.md) 等、既存ADRが定めた設計を破る変更は、既存ADRをSupersedeする新規ADRなしに行わない。
3. **1つのPull Requestは1つの責務のみを変更する**。「ついでの修正」「関係ないリファクタリング」を同じPRに混ぜない。複数の責務にまたがる変更が必要な場合は、PRを分割する。
4. **大きな関数を禁止する**。目安として、1関数あたり最大30文（statements）・最大分岐数8・最大循環的複雑度8・最大引数5を上限とし、`pyproject.toml` の `[tool.ruff.lint]`（`C90`, `PLR0915`, `PLR0912`, `PLR0913` 等）で機械的に検出する（[ADR-0002](0002-lint-format-typecheck-tooling.md) のツール選定に基づく）。閾値内に収まっていても、複数の責務を持つ関数は分割する。
5. 上記の規律は、`docs/adr/0009-ai-agent-operating-policy.md` に基づき、AIコーディングエージェントによる実装にも人間の実装と同様に適用される。

## 検討した代替案

- **規律をドキュメントのみで運用し、機械的な強制はしない**: コードレビューの負荷に依存することになり、レビュアーが変わると基準がぶれる。関数サイズについては機械的に検出可能なため、lintルールとして強制する方針とした。
- **PRサイズ（行数）で1PR1責務を強制する**: 行数は責務の単一性の代理指標としては不正確（大きなデータ追加PRは行数が多くても単一責務であり得る）であるため、機械的な強制は行わず、レビューで判断する運用とした。

## 結果（トレードオフ）

- 開発速度は短期的に低下する（PR分割・関数分割の手間が増える）。これは意図的なトレードオフであり、長期保守性を優先する本プロジェクトの目的に合致する。
- 関数サイズの閾値は初期見積もりであり、実装を進める中で非現実的だと判明した場合は、本ADRを更新せず `pyproject.toml` の設定値調整のみで対応してよい（閾値の微調整は設計原則そのものの変更ではないため）。ただし閾値を大幅に緩める場合は、その理由をコミットメッセージに明記する。
