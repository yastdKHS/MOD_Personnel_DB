# CLAUDE.md — Claude Code 向け作業規約

このファイルは Claude Code（本リポジトリで作業するAIコーディングエージェント）が読む、リポジトリ固有の運用ルールです。人間向けの一般的な開発ガイドは `CONTRIBUTING.md`、全AIエージェント共通の規約は `AGENTS.md` を参照してください。本ファイルは `AGENTS.md` を上書きしません。両方に従ってください。

## プロジェクトの本質

これは「PDFをパースするスクリプト」ではなく、「10年以上、複数の担当者・複数のAIセッションをまたいで保守され続けるデータ基盤」です。以下を常に優先してください。

1. **実装速度より設計品質を優先する** — 動くことより、既存ADRが定めた設計に従っていることを優先する（[ADR-0014](docs/adr/0014-development-discipline.md)）。
2. **正しさより先に、間違いに気づける設計** — 新しいパース処理やデータ変換を書くときは、必ず検証（validation）とテスト（できればゴールデンファイル）を伴わせる。
3. **ハードコード禁止領域** — PDFのレイアウト依存の値（座標・列位置・見出し文字列など）を `src/` 内のPythonコードに直接埋め込まない。必ず `layouts/` のレイアウト定義を経由する。理由は [ADR-0003](docs/adr/0003-layout-definition-strategy.md)。
4. **ドメイン知識はコードではなくデータ** — 階級名の表記ゆれ、組織名の改称履歴、氏名の異体字などは `src/` のロジックではなく `knowledge/` のデータファイルとして表現する。

## 中核パイプラインは変更禁止

処理の中核パイプラインは以下の6段階に**固定**されている（[ADR-0011](docs/adr/0011-fixed-core-pipeline.md)）。この構成・順序・名称を変更する実装は、明示的な指示がない限り行わない。

```
Document Analyzer → Layout Detector → Section Parser → Field Extractor → Normalizer → Validator
```

新しい様式・表記ゆれ・例外への対応は、この段階構成を変えるのではなく、以下の優先順位でデータ側に反映する（[ADR-0012](docs/adr/0012-error-handling-priority-order.md)）。

1. `knowledge/` への追加（表記ゆれ・別名・改称履歴等）— 正規表現の追加より常に優先する
2. `layouts/` への追加（様式・構造の差異）— `src/` の例外処理より常に優先する
3. `src/` 内の例外処理・正規表現による特殊対応 — 上記2つで表現できない場合の最後の手段。追加時は理由をコメント・PR説明に明記する

検証NGや事後的な誤り修正は、単なる修正ログではなく `knowledge/learning_dataset/` にLearning Datasetとして記録する。「いつ・誰が・何を直したか」だけでなく、「どの段階で・なぜ誤ったか・Knowledge Base/Layoutへの反映につながったか」を残す（[ADR-0013](docs/adr/0013-learning-dataset-not-correction-log.md)）。

## コードの粒度

- **大きな関数を禁止する**。目安は1関数あたり最大30文・最大分岐数8・最大循環的複雑度8・最大引数5。`pyproject.toml` のruff設定（`C90`, `PLR09xx`）で機械的に検出されるが、閾値内でも複数責務を持つ関数は分割する（[ADR-0014](docs/adr/0014-development-discipline.md)）。
- **1つのPull Requestでは1つの責務のみを変更する**。ついでのリファクタリング・無関係な修正を混ぜない。複数箇所にまたがる変更が必要な場合はPRを分割する。

## 作業前に確認すること

- 大きな設計変更（データモデル、パイプライン段階、技術選定）を行う場合は、先に `docs/adr/` に新しいADRを追加してから実装する。ADRのフォーマットは `docs/adr/README.md` を参照。特に中核パイプライン（[ADR-0011](docs/adr/0011-fixed-core-pipeline.md)）を変更する提案は、通常のADR追加だけでなく、必ず事前にユーザーへ確認する。
- 既存のADRと矛盾する変更をする場合は、既存ADRを "Superseded" にして新しいADRを追加する。ADRを無断で書き換えて履歴を消さない。
- `pyproject.toml` がツール設定の唯一の情報源。lint/format/型チェック/テスト設定を他ファイルに重複して作らない。

## 禁止事項

- 実在の人事発令PDF（サンプル目的以外）や、それらから抽出した個人データをリポジトリにコミットしない。サンプルは `sample_pdfs/README.md` の基準を満たすもののみ。
- `logs/`・`tmp/` 配下の内容をコミットしない（`.gitignore` で除外済みだが、意図的な `git add -f` も行わない）。
- `git push --force`、`git reset --hard`、ブランチ削除などの破壊的操作は、明示的な指示がない限り行わない。
- pre-commit フックやCIのチェックを `--no-verify` 等でバイパスしない。失敗したら原因を修正する。
- `knowledge/` 配下のデータを一括置換・自動生成で書き換えない。これは人手でレビューされるべき正データベース（reference data）であり、変更は差分が追跡・説明可能な単位で行う。

## テスト・検証

- 実装後は `pytest` を実行し、特に `tests/golden`（`sample_pdfs/` → `sample_outputs/` の比較）を壊していないことを確認する。
- `sample_outputs/` のゴールデンファイルを更新する場合、それは「正解を変更する」行為であるため、変更理由をコミットメッセージ・PR説明に明記する。無言で「テストを通すため」に更新しない。

## コミュニケーション規約

- ドキュメント・コミットメッセージ・コードコメントは日本語を基本とする（技術識別子・ライブラリ名等は英語のままでよい）。
- 不明な設計判断がある場合は、憶測で実装を進めず、ユーザーに確認する。特にデータモデルの破壊的変更、依存ライブラリの新規追加、公開データの取り扱い範囲の変更は要確認。
